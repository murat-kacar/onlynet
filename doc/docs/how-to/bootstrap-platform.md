# Bootstrap The Platform

This guide describes the **first-run** procedure that takes a fresh host to
a state where the platform host can serve `/login` and the platform worker
can claim `tenant.create` jobs.

It runs once per deployment, before any tenant exists. It is the
prerequisite step for [`./provision-tenant.md`](./provision-tenant.md).

## Invariants After Bootstrap

These statements MUST hold once the procedure completes:

1. PostgreSQL 17 is reachable on the platform host.
2. Database `tabflow_platform` exists, owned by role
   `tabflow_platform_app`, per
   [`../reference/database/schema.md`](../reference/database/schema.md#naming-convention).
3. The role `tabflow_platform_app` has `LOGIN` and table-level rights on
   `tabflow_platform` only â€” no superuser, no cross-database access.
4. Platform migration history is current
   ([AD-0008](../reference/architecture/decisions.md#ad-0008-ef-core-as-schema-and-migration-authority)).
5. Exactly one Identity user exists in the platform database, with the
   `owner` role per the platform identity baseline in
   [`../reference/database/schema.md`](../reference/database/schema.md#platform-identity).
6. The bootstrap admin's password was generated by the bootstrap command,
   shown exactly once on stdout, and is not stored anywhere on disk. Only
   the password hash persists.
7. First successful login by the bootstrap admin forces a password change.

## Inputs

Required:

- Bootstrap admin email (e.g. `ops@cafetech.uk`)
- PostgreSQL superuser credentials (used only during bootstrap, never
  written into TabFlow configuration)

## Steps

### 1. Verify PostgreSQL 17

PostgreSQL administration is operator-owned per
[AD-0007](../reference/architecture/decisions.md#ad-0007-postgresql-17-as-the-storage-baseline).
PostgreSQL 17 MUST be installed and running before continuing.

### 2. Create The Platform Role And Database

Run as the PostgreSQL superuser:

```sql
CREATE ROLE tabflow_platform_app
  WITH LOGIN
       PASSWORD '<generated-strong-password>'
       NOSUPERUSER NOCREATEDB NOCREATEROLE NOREPLICATION;

CREATE DATABASE tabflow_platform OWNER tabflow_platform_app;
```

The role MUST NOT have `SUPERUSER`, `CREATEDB`, `CREATEROLE`, or
`REPLICATION` rights. It MUST own only `tabflow_platform`; cross-database
ownership violates the platform/tenant boundary in AD-0001.

The generated password lives only in the platform host's host-owned
configuration source (e.g. systemd `EnvironmentFile=` or
`appsettings.Production.json` outside the source tree per
[`./deploy-to-production.md`](./deploy-to-production.md)).

### 3. Configure The Platform Host Connection String

Write the connection string to the host-owned configuration:

```
ConnectionStrings__PlatformDb=Host=localhost;Database=tabflow_platform;Username=tabflow_platform_app;Password=<generated-strong-password>
```

### 4. Apply Platform Migrations

```bash
dotnet ef database update \
  --project src/infra/postgres/TabFlow.Migrations.csproj \
  --context PlatformDbContext \
  --connection "$ConnectionStrings__PlatformDb"
```

After this step, `tabflow_platform` MUST contain the platform identity
tables (`AspNetUsers`, `AspNetRoles`, etc.), the tenant registry tables,
the provisioning job tables, the platform audit log, and
`__EFMigrationsHistory`.

### 5. Create The Bootstrap Admin

The bootstrap admin is **never** created by an EF Core migration with a
hard-coded password hash. Migration-time hashes are static, leak through
git history, and cannot be rotated without a new migration. The admin is
created by a one-shot CLI command that uses the same `UserManager`
primitives the rest of the platform uses.

```bash
dotnet run --project src/apps/platform/TabFlow.Platform.csproj -- \
  bootstrap-admin \
  --email <bootstrap-admin-email>
```

Behaviour:

1. The command refuses to run if any user already exists in `AspNetUsers`.
2. It generates a high-entropy password using a CSPRNG.
3. It calls `UserManager.CreateAsync` with the generated password â€” the
   resulting hash uses ASP.NET Core Identity's current
   `IPasswordHasher` defaults.
4. It assigns the `owner` role.
5. It stamps a `tabflow:must_change_password` claim on the user so the
   first authenticated request after sign-in is redirected to
   `/change-password` by `PasswordChangeRequiredMiddleware`. The claim
   is removed by the `ChangePassword` page on a successful rotation.
6. It writes a row to `platform_audit_log` with `action = 'auth.bootstrap'`.
7. It prints the generated password to stdout exactly once and exits.

The operator captures the printed password, signs in at
`https://<platform-host>/login`, and is forced through `/change-password`
on the first authenticated request.

### 6. Smoke Verify

```bash
curl -fsS https://<platform-host>/health/ready
psql -h localhost -U tabflow_platform_app -d tabflow_platform \
  -c 'SELECT count(*) FROM "AspNetUsers";'
```

`/health/ready` MUST return `200`. The user count MUST be exactly `1`.

The platform host is now ready to serve admin traffic. The platform
worker is the next start-up; it polls the platform DB for provisioning
jobs and is what the next how-to depends on.

## Failure Modes

- **Bootstrap admin already exists.** By design. The command refuses to
  run twice. Re-running on a healthy platform requires the deliberate
  recovery procedure in [`./rotate-secrets.md`](./rotate-secrets.md), not
  a second bootstrap.
- **Migration history mismatch.** The migrations project snapshot
  disagrees with the model. Resolve at the migrations source (regenerate
  via the design-time factory per AD-0009); never hand-edit
  `__EFMigrationsHistory`.
- **Role lacks the right privileges.** The `tabflow_platform_app` role
  was created with `SUPERUSER` or owns objects across databases.
  Recreate the role per Step 2; use `REASSIGN OWNED` if the role already
  owns the wrong objects.

## Re-Bootstrap

The `bootstrap-admin` command refuses to run twice on a populated
database. Recovery from a lost admin credential is a separate procedure
covered in [`./rotate-secrets.md`](./rotate-secrets.md); it is not a
re-bootstrap.

## Anti-Patterns

- Creating the bootstrap admin with `INSERT INTO AspNetUsers` and a
  hand-crafted hash. Identity hash format is hasher-version-dependent and
  brittle across .NET releases.
- Reusing PostgreSQL superuser credentials in
  `ConnectionStrings__PlatformDb`.
- Naming the platform database anything other than `tabflow_platform`.
- Using a development name such as `tabflow_demo` in production.
- Persisting the generated bootstrap password to a file or environment
  variable. Capture it from stdout once.
